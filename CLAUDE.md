## Code conventions

- **Double quotes** for string literals in all TS/JS source (whole monorepo, including `apps/web`). No ESLint quote rule enforces single â€” safe.
- **No comments on Prisma schema models.** Field-level `//` comments forbidden; rationale lives in `docs/`. Migration SQL `--` comments are fine.
- **Branch names + commit messages + PR titles + spec docs are shared artifacts** â€” no EPIC-XX or L1-L9 labels in them. Use feature names. EPIC labels stay in `reference/todo/` only.
- **One-liner commit messages.** No body. No `Co-Authored-By` trailer. Conventional Commits format (`feat(api): ...`, `fix(db): ...`, etc.).
- **Subpath exports for `@elorae/db` pure helpers.** Client-component imports must use `@elorae/db/color` or `@elorae/db/pantone` (NOT the main barrel) to avoid dragging Prisma/mariadb into the client bundle.

## Architecture nuances worth flagging

- **VPS MariaDB is shared between local dev and the VPS-deployed web.** A local destructive query (delete, truncate, bad migration) hits the same data the client demo shows. Run migrations only via `pnpm -F @elorae/db migrate:deploy`; never `migrate dev --create-only` without thinking. **On prod, the deploy workflow's `migrate` job auto-applies pending migrations on every push to master (unconditional â€” `needs: pull`, blocks deploy on failure), so no manual `migrate:deploy` post-merge. It does NOT seed** â€” RBAC/permission rows are seeded by hand (surgical SQL). Don't conflate the two. No automatic backups â€” set up daily `mariadb-dump` â†’ R2 separately (TODO).
- **`Item` is dual-owner** (web for ERP forms; api for Jubelio catalog ingest). All api writes go through `@elorae/db/item-writer.ts` helpers, stamping `source = JUBELIO_INGEST`. See `docs/BOUNDARY.md Â§3.3`.
- **`StockAdjustment` is dual-owner** (web for ERP-driven adjustments; api for Jubelio stock webhooks). Api writes go through `@elorae/db/stock-writer.ts` `applyJubelioStockAdjustment`, stamping `source = JUBELIO_WEBHOOK` + `idempotencyKey`. See `docs/BOUNDARY.md Â§3.1`.
- **Jubelio webhook signature is HMAC-SHA256 with header `Sign`** â€” not plain SHA256 with `webhook-signature`. Jubelio's docs *text* is wrong; their code example is correct. See `docs/BOUNDARY.md Â§9 Q1`.
- **Catalog ingest pulls from Jubelio** via `POST /jubelio/catalog/sync` (apps/api). Product push (ERP â†’ Jubelio) is wired via the outbox â€” see sub-3 below.
- **Inbound webhook pipeline ships in BullMQ-backed queue.** `JubelioWebhookEvent` carries authoritative status (`RECEIVED â†’ PROCESSING â†’ PROCESSED / SKIPPED / DEAD`). Worker runs in apps/api process, concurrency 4 by default (`JUBELIO_WORKER_CONCURRENCY`). Sweeper rescues stuck rows every 10 min. Note: shared-state writers (e.g. `InventoryValue` aggregates) must use atomic Prisma `increment`/`decrement`, never read-modify-write â€” the worker is concurrent and the ERP-ship path races the ship-webhook across processes.
- **Outbound push pipeline ships in `JubelioOutbox` table + `outbox-poller` + `outbox-processor`.** Handlers in `apps/api/src/jubelio/outbox/handlers/`: `product_push`, `stock_push`, `salesorder_pick`, `salesorder_pack`, `salesorder_ship`. Already-in-state Jubelio responses are skipped (not retried). New push types add: handler file + payload builder + spec + router case.
- **Marketplace stock is reserve-at-ingest, consume-at-ship, release-on-cancel.** `StockReservation` ledger (one row per `salesorderDetailId`) + aggregate `InventoryValue.reservedQty`, driven by `reserveOrder`/`consumeOrder`/`releaseOrder` in `@elorae/db/reservation-writer.ts`. `available = qtyOnHand - reservedQty`, derived at read time â€” never stored. `consumeOrder` deducts `qtyOnHand` via a `StockAdjustment` stamped `source = FULFILLMENT_CONSUME`, and is idempotent so the Jubelio ship webhook and the ERP Ship button can both trigger it safely. Jubelio stock push sends `available`, not raw `qtyOnHand`. See `docs/BOUNDARY.md` D6.
- **PWA service worker gotchas** (cost hours 2026-07-06). (1) `apps/web/proxy.ts` IS the Next 16 middleware â€” `proxy.ts` is Next 16's rename of `middleware.ts` (compiles to `.next/server/middleware.js`, runs on every request); it is NOT dead code. Its allow-list gates `/pwa/*`; the Serwist SW script `/pwa/sw.js` is currently gated (307 â†’ /login unauth) â€” a SW script behind auth is wrong (breaks update fetches). (2) The `/pwa` Serwist SW (`app/pwa/sw.ts` â†’ `public/pwa/sw.js`) only exists in a **production build** (`next build --webpack`; disabled in dev, and Turbopack skips the plugin per PR #92) â†’ offline behavior is testable **only on prod** (local `next start` over http has a NextAuth secure-cookie login quirk). (3) **Never commit next-pwa/serwist generated files** (`public/sw.js`, `public/workbox-*.js`, `public/pwa/sw.js`, gitignored) â€” a stale committed `workbox-<hash>.js` rides the Docker `COPY` into the build, poisons the SW precache manifest, then 404s at runtime â†’ SW install fails â†’ offline dinos (root cause of the #113 field-test failure; fixed #115). (4) Root `/` next-pwa SW (scope `/`, caches `/^https?.*/`) and the `/pwa` Serwist SW coexist.

## What NOT to do

- Don't put EPIC-XX refs in commits, PRs, branch names, or shared specs.
- Don't bundle multiple sub-projects into one PR. Each sub-project is its own slice.
- Don't write to web-owned tables from apps/api (and vice versa) without going through a `@elorae/db` helper â€” see `docs/BOUNDARY.md Â§3`.
- Don't add Prisma model comments. Don't add `Co-Authored-By` trailers to commits.
- Don't run `prisma migrate dev` against the shared VPS MariaDB â€” that creates throwaway migrations and resets state. Use `migrate:deploy` only.
- Don't deploy apps/api or apps/web to Vercel. Both need persistent processes; Vercel functions don't fit (node-cron, BullMQ workers). Production target is the Hostinger VPS; local-ngrok is the dev/demo fallback.
- **Don't run the whole test suite** (`pnpm -F @elorae/api test` with no filter, or repo-wide). It's slow + wasteful. Scope to the specs you changed: `pnpm -F @elorae/api test -- <pattern> [<pattern>...]` (jest treats each positional arg as a testPathPattern, OR-ed). Only widen if a change plausibly affects unrelated specs.
- **Don't type-check the whole repo** (turbo/all-package). Scope to the one package you changed: `pnpm -F @elorae/api type-check` only. (`@elorae/web` type-check stays the user's â€” it saturates disk; never run it.) tsc can't do single-file, so package-scope is the finest partial available.
